# 用好 AI 助手的操作手册（爸爸写给你）

> 这不是讲大道理的文章，是**照着做就能用**的步骤。
> 每一节都给你具体怎么操作、可以直接抄的模板。看不懂的问我。

---

# 第一部分 · 干活的规矩（每条都有好例子和坏例子）

## 规矩 1：派活说清楚

**❌ 坏**："做个游戏" —— 它只能瞎猜。

**✅ 好**："做一个第一人称射击游戏，能联网一起玩，打中敌人掉血"

想做好几件事，一次列出来：**"帮我做 1、2、3、4"**，它会一口气做完，不用你催。

## 规矩 2：让它找"真原因"，别让它"盖住问题"

**❌ 坏**：游戏卡了，它说"我加个东西重置一下就好"——这是把问题藏起来，病根还在。

**✅ 好**：你说**"先别修，告诉我到底为什么卡"**。它查清楚了再动手。

> 真事：游戏"打到第 5 个敌人就卡"，逼它找原因，挖出来是三个问题叠在一起。
> 要是当初"加个东西盖住"，一个都没解决。

👉 **说不清"为什么坏"的修复，都是假的。**

## 规矩 3：别信"应该好了"

**❌ 坏**：它说"这样**应该**就好了"——"应该" = 它自己没试过。

**✅ 好**：要求它**"改完自己跑一遍，把通过的结果贴给我看"**。
重现不了的 bug，让它老实说"你帮我打开看看截个图"。

## 规矩 4：用测试证明能用（两种测法都要）

**测法一 · 命令行自动测**：加了新功能，让它**顺手写个小测试自动检查**。
> 加了"手雷"，它就写测试：中间炸一下，查正中伤害最大、边上一半、范围外没伤害。
> 以后谁改坏了，一跑就报警。

**测法二 · 直接开网页测**：命令行过了不代表网页没问题。让它**真打开网页、截图、看报错**。
> 中文变方块、列表空白这种 bug，只有开网页才看得到。

## 规矩 5：开多个窗口一起干

AI 助手可以同时开好几个，像派好几个帮手。

**✅ 好做法（真这么干的）**：
- 主窗口写新功能，**另开一个窗口专门跑测试、改测试**。
- 遇到难 bug，**单独派一个助手去"侦查"**——专门翻代码找原因，回来报告。

👉 活多就分开几个窗口并行，省一半时间。

## 规矩 6：检查它（别全信）

**❌ 坏**：它分析得头头是道说"是这里重复了"，你就信。
（真事：让另一个助手去翻代码，发现根本没重复，它说错了。）

**✅ 好**：**真相在代码里，不在它嘴上。** 重要的地方自己打开文件看那一行。
它说"测试过了"，就看它有没有真**贴出"通过"的结果**——没结果就是没测。

👉 可以信任，但重要的地方自己核实一眼。

# 第二部分 · 四个文件怎么建、怎么维护（重点）

AI **每次重新聊天就忘光**上次干的事。解决办法：把工作状态写进文件，
每次开始让它先读。这四个文件是核心。

## 怎么建？三步

**第 1 步**：项目根目录建一个 `CLAUDE.md`（这是"总规矩"，AI 每次自动读）。

**第 2 步**：建一个 `.agent/` 文件夹，里面放三个工作文件：
```
.agent/todo.md          要做的事
.agent/codexreview.md   要修的问题
.agent/test.md          测出来的 bug
```

**第 3 步**：在 `CLAUDE.md` 里写一句话告诉 AI："每次新对话先读这三个文件"。

---

## 文件一：`CLAUDE.md`（总规矩 —— 最重要）

放在项目根目录，AI 每次**自动读**。把你对它的所有硬性要求写这。
**可以直接抄下面这个模板，改成你自己的：**

```markdown
# 我的项目 — AI 工作约定

## 每次新对话先做
先读 `.agent/todo.md`、`.agent/codexreview.md`、`.agent/test.md`，
不要凭记忆做事。

## 改 bug 的硬性要求
1. 不要盲目加"保险"盖住症状 —— 要找根本原因
2. 改完自己跑测试验证，把结果贴出来
3. 不要谎报修好 —— 没重现的 bug 不许说"应该好了"
4. 跟参照项目比对 —— 如果别人没这毛病，就是我们自己的 bug

## 测试入口
（写上你的测试命令，比如）
    bash tests/run_all.sh

## 项目现状
（一句话写当前做到哪了，让 AI 快速进入状态）
```

👉 **维护方法**：每次定下新规矩、或者发现 AI 老犯同一个错，就往这加一条。
这文件越准，AI 越听话。

---

## 文件二：`.agent/todo.md`（要做的事）

按重要程度排队，做完打勾。**抄这个模板：**

```markdown
# 要做的事

## P0 — 最急（挡路的 bug，先修）
- [ ] xxx 坏了，玩家没法玩

## P1 — 重要
- [ ] 加 xxx 功能
- [x] 已经做完的（打勾保留，方便回头看）

## P2 — 有空再做
- [ ] xxx 优化

## 已完成（保留，方便回溯）
- [x] xxx —— 2026-XX-XX 完成，怎么做的
```

👉 **维护方法**：
- 你提需求 → 写进对应优先级（P0 最急，P2 不急）
- AI 做完 → 让它把 `[ ]` 改成 `[x]`，**别删**，移到"已完成"段
- 每隔一阵清理一下，把过时的删掉

---

## 文件三：`.agent/codexreview.md`（要修的问题）

别人（或别的工具）帮你检查代码挑出的毛病，贴这里，让 AI 一条条修。

```markdown
# 代码检查清单

## 2026-XX-XX 检查发现

### P0 严重
- xxx 有安全漏洞，在 文件名:行号

### P1 一般
- xxx 写法不好，建议改成 yyy

---
（AI 每修一条，在下面追加：）
### [x] 已修复 — 改了 文件名:行号，怎么改的
```

👉 **维护方法**：让 AI 每修一条**追加一段"已修复"说明改了哪里**，原文留着。
这样你能核对它到底修没修、修对没。

---

## 文件四：`.agent/test.md`（测出来的 bug）

你自己玩的时候发现的问题、报错日志，贴这里。**抄这个格式：**

```markdown
# 测试发现的问题

## 2026-XX-XX — 一句话描述

**症状**：（看到了什么不对）
**重现步骤**：
1. xxx
2. xxx
**日志**：
（把报错粘进来）
**期望**：（本来应该怎样）

---
（AI 修完追加：）
### [x] 已修复 — 根本原因是 xxx，改了 文件名:行号
```

👉 **维护方法**：贴 bug 时**尽量给重现步骤 + 截图/日志**，AI 才好找原因。
修完保留原文，下次类似 bug 能翻出来对比。

# 第三部分 · 日常怎么操作（一步步）

**开始一段工作**：
1. 打开 AI 助手
2. 说一句"读一下 todo 和 test.md"，让它进入状态
3. 说今天要干啥（具体！）

**做一件事的循环**：
1. 派活（说清楚）
2. 它做的时候看着，它问你就答
3. 做完让它**跑测试 + 贴结果**
4. 重要的改动自己扫一眼
5. 让它把 todo 里这条打勾 `[x]`

**发现 bug**：
1. 截图 / 复制报错
2. 贴进 `test.md`，写上重现步骤
3. 跟 AI 说"读 test.md，找根因修一下"（强调**找根因**）

**收工前**：
- 让它把"没做完的、为什么没做"列出来，你才知道真实进度
